App优化之启动速度、瘦身、耗电处理

Posted on 2019-11-23

耗电处理

  • 耗电的来源
    • CPU处理 processing
    • 网络Networking
    • 定位 Location
    • 图像 Graphics
耗电的优化
  • 降低CPU/GPU的功耗
  • 少用定时器
  • 优化I/O操作
    • 不要频繁写入小数据，最好批量一次性写入
    • 读写大量数据使用dispatch_io
      • 提供了基于GCD的异步文件I/O的API
      • 系统会优化磁盘访问
    • 数据量比较大，使用数据库（SQLite、CoreData）
  • 网络优化
    • 压缩数据
    • 缓存（多次请求结果相同的时候）
    • 断点续传（解决网络不稳定时可能多次传输相同内容）
    • 状态检测（网络不可用的时候，不要执行网络请求）
    • 超时交互（让用户取消长时间请求、超时请求）
    • 批量传输
      • 视频流，要整个文件或者块为单位下载，不要小数据包下载
      • 广告，一次性多下载一些，再多次慢慢展示
      • 邮件，一次下载多封，不要一封一封的下载
  • 定位：CLLocatoinManager
    • 如果只是需要快速确定用户位置，最好用CLLocationManager的requestLocation方法。
      • 定位完成后，会自动让定位硬件断电
    • 尽量降低定位精度，比如尽量不要使用精度最高的kCLLocationAccuracyBest
    • 需要后台定位时，尽量设置pausesLocationUpdatesAutomatically为YES，如果用户不太可能移动的时候系统会自动暂停位置更新
    • 尽量不要使用startMonitoringSignificantLocationChanges，优先考虑startMonitoringForRegion:
    • 如果不是导航应用，尽量不要实时更新位置，定位完毕就关掉定位服务
  • 硬件监测优化
    • 用户移动、摇晃、倾斜设备，产生动作（motion）事件
    • 触发设备的加速计、陀螺仪、磁力计
    • 不用的场合及时关闭这些硬件

App启动优化

App启动的三个阶段
  1. dyld
  2. runtime
  3. main

  • dyld

    • 减少动态库、合并一些动态库（定期清理不必要的动态库）
    • 减少Objc类、分类的数量、减少Selector数量（定期清理不必要的类、分类）
    • 减少C++虚函数数量
    • Swift尽量使用struct

  • runtime

    • 文件内容的解析
    • +load
    • 各种objc的初始化（注册Objc类、初始化对象等）
    • C++静态初始化器
    • 用+initialize方法和dispatch_once取代所有的__attribute__((constructor))、C++静态构造器、ObjC的+load

  • main

    • 按需加载
    • 在不影响用户体验的前提下，尽可能将一些操作延迟，不要全部都放在finishLaunching方法中

安装包瘦身

  • 安装包（IPA）主要由可执行文件、资源组成
  • 资源（图片、音频、视频等）
    • 图片资源放到 App assets里，选择默认的压缩类型，相比放到bundle，压缩大小能减少15%~20%，解码时间减少20%（High Efficiency Image File Format）
    • 采取无损压缩
    • 去除没有用到的资源：https://github.com/tinymind/LSUnusedResources
  • 可执行文件瘦身
    • 编译器优化
    • Strip Linked Product、Make Strings Read-Only、Symbols Hidden by Default设置为YES
    • 去掉异常支持，Enable C++ Exceptions、Enable Objective-C Exceptions设置为NO， Other C Flags添加-fno-exceptions
    • 利用AppCode检测未使用的代码：菜单栏 -> Code -> Inspect Code
  • 编写LLVM插件检测出重复代码、未被调用的代码
  • LinkMap
    • 生成LinkMap文件，可以查看可执行文件的具体组成(每个类最终占用可执行文件的大小)
    • Build Settings -> Linking -> Path to Link Map File && Write Link Map File(YES)
      一些好的开发习惯
  • 不要为了一片树叶引入一片森林
  • check引入资源的大小
  • 及时清理废弃模块
  • 同质库只引入一种
    • 譬如AFnetworking vs ASIHttpRequest
  • 建立预警机制
    • 脚本打包，除了正常的生成ipa包之外，也要生成分析文件，列出相对上一次上线包大了多少，类文件增加了多少
