组件化-iOS组件通信方案

Posted on 2019-11-23

业务模块化架构设计，可以达到明确模块职责及边界，提升代码质量，减少复杂依赖，优化编译速度，提升开发效率等效果

设计模式 vs 架构设计

App 架构设计更多强调的是分层，基于设计模式六大原则之一的单一职责原则，将系统划分为基础层，网络层，UI层等等，以便于维护和扩展.

设计模式的范畴比架构设计小，强调的是类与类之间的关系。
而架构可大可小，小到类之间的关系、大到模块之间、CS之间的设计。

1. 架构分层设计

以上是快乐平安App的主要架构，主要划分为4层，目前有3个项目共用了底层库。

  • 基础层：
    • 这里放一些基础的、与业务无关的公共组件，尽量用第三方的通用组件。
  • 基础业务层：
    • 这里放一些基础能力支撑的库。基于这一层，上层的业务组件可以⽅便地开发⾃己的组件。
    • 注: 需要注意，这里只提供基础能力支持，不提供界面UI相关的功能。另外，被多个业务组件使 ⽤的基础业务组件才放这里。
  • 框架层:
    • 这里提供组件管理、路由管理功能。通过框架层，用户可以对基础业务组件、业务组件进行拼装、组合。框架层提供工程模板，通过简单的配置，即可生成一个壳工程，并加入⾃己的业务。
    • 框架层还提供JSBridge功能，并设计支持weex、flutter等运行框架。
    • 框架层会放入平台所有组件的索引Spec。为了使用所有这些组件，Podfile中放入:
  • 业务层
    • 业务层放独立的业务组件。业务组件之间通过框架层进行通信。
    • 注: 基础层可以直接由上层调用。业务层和基础业务层的调用，都要通过框架层的组件管理进⾏通信。不允许直接调用业务层、基础业务层内的组件。

2. 组件通信

除了基础层可以直接调用外，以上层的组件间调用，需要经过组件管理、或组件路由。

YYModule 是实现iOS组件化的一套库。 主要包含两部分内容:

  • YYModuleManager
    • 基于protocol，使⽤接⼝注册的方式，实现组件间解耦，是主要的组件化方案。
  • YYRouter
    • 添加url路由支持，主要用于app间的调用、web跳转、通知跳转等场景，作为对基于protocol的组件化方案的补充。

YYRouter 基于MGJRouter ，做了了如下改动:

  1. 去掉了 objectForURL 相关方法，统一调用方式。
  2. 路由处理流程中添加了interceptor，⽀持灵活地定制、处理路由。

4. 通信方案优缺点分析

以上是一些具体的实战应用，通信组件经过了特定需求的改造。为了更全面的分析方案优劣，我们一般会对如下几种方案进行考量：

  1. 基于面向协议思想的服务注册方案
  2. 基于路由 URL 的 UI 页面统跳管理
  3. 基于反射的远程接口调用封装
  4. 基于通知的广播方案
基于面向协议思想的服务注册方案。
  • 优势
    • 代码自动补全和编译时检查都有效，实现简单，调用方便
    • 协议的所有实现仍然在模块内部，不需要写反射代码
    • 依赖接口而不是依赖对象，符合团队协作的“面向协议编程”的思想
  • 劣势
    • 如果服务提供方和使用方依赖的是公共模块中的同一份协议（protocol）, 当协议内容改变时，存在所有服务依赖模块编译失败的风险。
    • 需要一个注册过程，将 Protocol 协议与具体实现绑定起来。

业界里，蘑菇街的 ServiceManager 和阿里的 BeeHive 采用了这个方案。

例：

每个模块提供自己对外服务的协议声明，然后将此声明注册到中间层。调用方能从中间层看到存在哪些服务接口，然后直接调用即可。

基于路由 URL 的 UI 页面统跳管理。
  • 优势
    • URL 本身是一种跨多端的通用协议。使用路由URL统跳方案的优势是动态性及多端统一 (H5, iOS，Android，Weex/RN);
  • 缺点
    • 能处理的交互场景偏简单(所以一般更适用于简单UI页面跳转、app跳转）
    • 一些复杂操作和数据传输用这个方式实现，效果不高。

目前天猫和蘑菇街都有使用路由URL 作为自己的页面统跳方案，达到解耦的目的

基于反射的远程接口调用封装。
  • 优势
    • 实现原理是Target-Action
    • 调用简单方便，代码自动补全和编译时检查都仍然有效。
    • 相比于URL Router，`Target-Action也不需要注册和内存占用
  • 劣势
    • category 存在重名覆盖的风险，需要通过开发规范以及一些检查机制来规避。
    • Mediator 只是收敛了 hardcode, 并未消除 hardcode, 仍然对开发效率有一定影响。
    • 开发者定义的类和方法，必须要遵守Target-Action的命名规则，调用者可能会因为硬编码问题导致调用失败

业界的 CTMediator 开源库，以及美团都是采用类似方案。

正如上总结，这种方式存在大量的hardcode字符串、无法触发代码自动补全、容易出现拼写错误等问题，而且这类错误只能在运行时触发相关方法后才能发现。无论是开发效率还是开发质量都有较大的影响。

解决的思路可以借鉴「移动端向后台接口发网络请」这个案例。针对这类问题，创建一个网络层，将这类“危险代码”封装到里面。上层业务调用时网络层接口时，不需要 hardcode 字符串，也不需要理解内部麻烦的逻辑。

类似的，可以将模块间通讯也封装到一个“网络层”中（或者叫消息转发层）。这样危险代码只存在某几个文件里，可以特别地进行code review和联调测试。后期还可以通过单元测试来保障质量。模块化方案中，我们可以称这类“转发层”为 Mediator (当然你也可以起个别的名字)。同时因为 performSelector 方法附带参数数量有限，也没有返回值，所以更适合使用 NSInvocation 来实现。

然后各个业务模块依赖 Mediator, 就可以直接调用这些方法了。

基于通知的广播方案

基于通知的模块间通讯方案，实现思路非常简单, 直接基于系统的 NSNotificationCenter 即可。优势是实现简单，非常适合处理一对多的通讯场景。劣势是仅适用于简单通讯场景。复杂数据传输，同步调用等方式都不太方便。模块化通讯方案中，更多的是把通知方案作为以上几种方案的补充。
